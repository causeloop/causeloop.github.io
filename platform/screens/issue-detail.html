Auth service: JWT validation timing out under peak load — P1 incident

OPENED Jun 6, 2026 at 09:14 UTC UPDATED 2h ago SOURCE PagerDuty · INC-44821
AI Root Cause Analysis Causeloop AI Confidence 91%
Root cause

This incident is a direct recurrence of a known pattern (Auth service timeouts, first detected 43 days ago). The immediate trigger is JWT HMAC-SHA256 signing operations becoming CPU-bound at ≥8,000 concurrent sessions, causing the validation queue to back up beyond the 500ms timeout threshold.

The Cause

The auth service thread pool (maxThreads=200) is not sized for current peak load of 12,400 rps observed post-expansion. JWT key signing is not offloaded to a dedicated crypto worker — all signing happens on request threads, causing head-of-line blocking under load.

Contributing factors

1. No circuit breaker on the auth gateway — downstream services continue retrying failed auth calls, amplifying load by 3–4× during degradation.
2. Token refresh not distributed — a cohort of sessions refresh simultaneously (token expiry set uniformly to 1h after the last deployment), causing traffic spikes at predictable intervals.
3. Missing async key rotation — JWKS key rotation blocks the request path briefly during key reload, exacerbating timeouts during high load windows.

Why this keeps recurring

This is the 8th occurrence in 90 days (see loop detection below). Each prior incident was resolved by scaling up compute, without addressing the thread-pool architecture or synchronous signing path. Causeloop predicts a 93% probability of recurrence within 14 days unless the signing path is decoupled.
